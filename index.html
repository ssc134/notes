Time complexity glossary
Big-O/Ω/Θ, worst/best/average-case, and amortized time
📐 The mathematical notations (the bounds)
  • Big O (O): the upper bound. The algorithm’s time will grow no faster than this rate.
  • Big Ω (Ω): the lower bound. The algorithm’s time will take at least this long to grow.
  • Big Θ (Θ): the tight bound. The exact growth rate (runtime is sandwiched by this function).
  • Little o (o) & little ω (ω): strict upper and lower bounds (runtime must be strictly less than or greater than the bound, never equal).
⏱️ The scenarios (the inputs)
  • Worst-Case: the single specific input of size n that takes the absolute maximum time. The industry standard is to bound this using O.
  • Best-Case: the single specific input of size n that takes the absolute minimum time. Mathematically, this is best described using Ω.
  • Average-Case: the expected time across all possible inputs of size n, weighted by the probability of each input occurring.
🏦 Amortized complexity (the sequence)
  • Amortized Time: the guaranteed average cost per operation over a long sequence. Occasional expensive operations (like an array copying itself to resize) are “paid for” by a large number of cheap operations. It ignores probability.
Quick code snippets
Common patterns + caveats (Java)
Frequency counter (preferred)
hm.merge(i, 1, Integer::sum);
Caveats
  • Null values: if hm can contain a key mapped to null, merge treats it as “absent” and sets it to 1.
  • Type: works when the map value type is Integer (or compatible with the remapping function).
  • Concurrency: not atomic on a plain HashMap; use ConcurrentHashMap for safe concurrent updates.
Frequency counter (simple)
hm.put(i, hm.getOrDefault(i, 0) + 1);
Caveats
  • Double lookup: performs a getOrDefault and then a put (slightly more work than merge).
  • Null values: if the key exists but maps to null, getOrDefault returns null, and null + 1 will throw.
  • Concurrency: not atomic on a plain HashMap; concurrent updates can be lost.
Iterate a Map (entrySet)
for (Map.Entry<String, String> entry : hm.entrySet()) { System.out.println("Key: " + entry.getKey() + ", Value: " + entry.getValue()); }
Caveats
  • Efficiency: prefer entrySet() when you need both key and value (avoids extra lookups vs keySet() + get).
  • Order: iteration order depends on the map type (e.g., HashMap unordered, LinkedHashMap insertion/access order, TreeMap sorted).
  • Concurrent modification: structurally modifying the map while iterating can throw ConcurrentModificationException (unless using concurrent collections / iterator rules).
Java comparators
In Java, a custom Comparator lets you control how objects are ordered when sorting a list/array or using ordered structures like a PriorityQueue.
The golden rule of compare(a, b)
  • Negative (< 0): a comes before b.
  • Zero (== 0): a and b are considered equal in order.
  • Positive (> 0): a comes after b.
// Sort students by grade, highest to lowest PriorityQueue<Student> topStudents = new PriorityQueue<>( (s1, s2) -> Integer.compare(s2.grade, s1.grade) );
Note: Prefer Integer.compare(a, b) (or Long.compare, etc.) over subtraction (a - b) to avoid silent integer overflow with very large numbers.
Creating an array of lists
List<Integer>[] buckets = new List[10];
Caveats
  • Generic array: you cannot write new List<Integer>[10] — Java does not allow creating arrays of parameterized types. Use new List[10] and declare the variable as List<Integer>[] (or use @SuppressWarnings("unchecked") if needed).
  • Initialization: each slot is null until you assign a list (e.g. buckets[i] = new ArrayList<>();). Initialize before use to avoid NullPointerException.
